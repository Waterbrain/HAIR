# HAIR PCB Design Handoff

## What is HAIR?

HAIR (Home Assistant IR Admin) is a custom Home Assistant integration for learning, storing, and replaying infrared remote control signals. It uses ESPHome-based hardware as IR proxies: the ESP32 handles the physical IR transmit/receive, while the HAIR integration in HA provides the admin UI, signal fingerprinting, device management, and command replay.

The goal of this handoff is to design a purpose-built PCB that can be manufactured (JLCPCB, PCBWay, etc.) or offered to others as a HAIR-compatible IR transceiver module.


## Current Hardware Setup (Dev/Prototype)

We are currently using an ESP32-C3-DevKitM-1 development board with discrete IR components wired to GPIO pins. The ESPHome YAML config (`esphome/hair1.yaml`) defines the full firmware:

```yaml
esphome:
  name: hair1
  friendly_name: HAIR1

esp32:
  board: esp32-c3-devkitm-1
  framework:
    type: esp-idf

remote_transmitter:
  id: ir_tx
  pin: GPIO9
  carrier_duty_percent: 50%
  non_blocking: true

remote_receiver:
  id: ir_rx
  pin:
    number: GPIO8
    inverted: true
    mode:
      input: true
      pullup: true
  dump: all
  tolerance: 25%
  idle: 10ms

  on_pronto:
    then:
      - homeassistant.event:
          event: esphome.remote_received
          data:
            protocol: "PRONTO"
            code: !lambda 'return x.data;'

infrared:
  - platform: ir_rf_proxy
    name: HAIR1 TX
    id: ir_proxy_tx
    remote_transmitter_id: ir_tx
  - platform: ir_rf_proxy
    name: HAIR1 RX
    id: ir_proxy_rx
    receiver_frequency: 38kHz
    remote_receiver_id: ir_rx
```


## How the Signal Pipeline Works

Understanding the full signal pipeline is important for PCB design decisions (range, LED count, receiver sensitivity, etc.).

### Receive (RX) Path

1. An IR remote button press emits modulated IR light (typically 38kHz carrier).
2. The IR receiver module (e.g., TSOP38238) on the ESP32 demodulates the carrier and outputs a digital pulse train on GPIO8.
3. ESPHome's `remote_receiver` component captures the raw timing data.
4. The `on_pronto` trigger converts the raw timings into Pronto hex format.
5. The ESP32 fires a `homeassistant.event` (`esphome.remote_received`) over its native API connection to HA.
6. HAIR's `SignalMonitor` listens for these events on the HA event bus.
7. The `EventParser` extracts protocol, code, and raw_timings from the event.
8. An S/L (Short/Long) pulse-duration fingerprint is computed from the Pronto timing words. Each timing word is classified as S(hort) or L(ong) using a median-based threshold. This mirrors how real IR receiver ICs decode -- they only care about relative pulse duration, not exact microsecond timing.
9. A device-level fingerprint groups signals by remote (using carrier frequency + preamble S/L pattern).
10. Signals are stored in HAIR's signal store and displayed in the Sniffer UI.

### Transmit (TX) Path

1. User triggers a command from the HAIR UI (or a button entity, remote entity, etc.).
2. HAIR's `DeviceManager.async_send_command()` looks up the stored command.
3. `build_command()` creates a `ProntoCommand` (or `RawTimingsCommand`) wrapping the stored data.
4. HA's `infrared.async_send_command()` routes to the configured emitter entity.
5. The ESPHome `ir_rf_proxy` platform receives the command via HA's native API.
6. ESPHome's `remote_transmitter` component modulates the carrier and drives the IR LED(s) on GPIO9.

### Signal Format: Pronto Hex

HAIR stores all commands as Pronto hex strings. Pronto is a universal IR format:

- 4-word header: `0000 006D 0022 0002` (learned format marker, frequency word, burst pair counts)
- Frequency word `006D` = 38.028 kHz (computed as `1000000 / (word * 0.241246)`)
- Timing words: pairs of mark/space durations in units of the carrier period
- Default carrier: 38000 Hz (HAIR constant `DEFAULT_CARRIER_FREQUENCY`)

The S/L fingerprinting classifies each timing word against a threshold derived from the median of all timing values in that code. Words significantly above the median are "L" (long), below are "S" (short), and very large values (>10x median, typically end-of-signal gaps) are ignored.


## PCB Functional Requirements

### Must-Have

1. **ESP32 microcontroller** -- ESP32-C3 (RISC-V, low cost, WiFi) is what we're using. ESP32-S3 is an option if you want Bluetooth too, but C3 is sufficient. The firmware is ESPHome, so any ESPHome-supported ESP32 variant works.

2. **IR Transmitter (TX)** -- One or more high-power IR LEDs driven via a MOSFET/transistor on a GPIO pin.
   - Current GPIO: GPIO9
   - Carrier frequency: 38kHz (generated by ESPHome's `remote_transmitter`)
   - Carrier duty cycle: 50%
   - `non_blocking: true` -- TX runs in a task, so timing is handled by the ESPHome component
   - For good room coverage, consider 2-4 wide-angle IR LEDs (e.g., TSAL6200, VSLY5850) aimed in different directions, or a single high-power LED with a wide beam angle
   - A MOSFET driver (e.g., 2N7002, BSS138) is essential -- GPIO alone cannot source enough current for useful IR range

3. **IR Receiver (RX)** -- An integrated IR receiver/demodulator tuned to 38kHz.
   - Current GPIO: GPIO8 (inverted input, internal pullup)
   - Recommended part: TSOP38238 (38kHz, good noise rejection, widely available)
   - Alternative: VS1838B (cheaper, less noise rejection)
   - The receiver output is active-low, hence `inverted: true` in the ESPHome config
   - Tolerance is set to 25% in the ESPHome config for the `remote_receiver`
   - Idle timeout: 10ms (gap between bursts)
   - `dump: all` captures all protocols

4. **USB-C connector** -- For power and initial ESPHome firmware flashing. After first flash, OTA updates work over WiFi.

5. **3.3V power regulation** -- ESP32-C3 runs at 3.3V. If powered from 5V USB, need an LDO (e.g., AMS1117-3.3).

6. **WiFi antenna** -- Either PCB trace antenna or U.FL connector for external antenna. PCB antenna is fine for most rooms. The ESP32-C3 module (ESP32-C3-MINI-1) has a built-in PCB antenna.

### Nice-to-Have

7. **Status LED** -- A small visible LED (not IR) on a GPIO for visual feedback (connected, receiving, transmitting). ESPHome has a `status_led` component.

8. **Multiple TX LED directions** -- For 360-degree coverage, mount IR LEDs pointing in 3-4 directions. This is a big UX win -- users don't have to aim the device.

9. **Mounting options** -- Screw holes or adhesive pad area for wall/ceiling mounting. Ceiling mount gives best coverage for a multi-directional TX design.

10. **IR blaster port** -- A 3.5mm TRS jack or JST connector for an external IR emitter cable. Useful for enclosed cabinets where line-of-sight to the device isn't possible.

11. **Temperature sensor** -- A DHT22 or BME280 adds ambient temperature sensing since the device is already on the network. Costs almost nothing on the BOM.

12. **Reset button** -- Physical reset button connected to EN or a GPIO for factory reset / safe mode boot.


## GPIO Pin Assignments (Current)

| Function | GPIO | Notes |
|----------|------|-------|
| IR TX | GPIO9 | MOSFET-driven, 38kHz carrier |
| IR RX | GPIO8 | Inverted, internal pullup, TSOP38238 |
| Status LED | (unassigned) | Any free GPIO |
| Boot/Flash | GPIO9 | Shared with TX (ESP32-C3 strapping pin -- check datasheet) |

**Important ESP32-C3 note:** GPIO8 and GPIO9 are strapping pins on the C3. GPIO9 is the boot mode select pin (must be HIGH during boot for normal operation). The IR TX circuit must not pull this pin low during boot. A pull-up resistor and MOSFET gate resistor should handle this, but verify in your schematic.


## ESPHome Firmware Notes

The firmware is pure ESPHome YAML -- no custom C++ components needed. Key points for the PCB designer:

- **Framework:** `esp-idf` (not Arduino). This is required for reliable IR timing on the C3.
- **OTA:** Enabled. After initial USB flash, all updates are over WiFi.
- **API encryption:** Enabled. The ESP32 communicates with HA over an encrypted native API connection.
- **WiFi:** Supports SSID/password config via secrets. Fallback AP mode ("HAIR1 Fallback") for initial setup.
- **Captive portal:** Enabled for WiFi setup.
- **Infrared platform:** Uses `ir_rf_proxy` platform from the ESPHome `infrared` component (available in ESPHome 2026.x). This creates `infrared.*` entities in HA that HAIR uses as TX/RX endpoints.

The `on_pronto` trigger is the RX event bridge: it converts received IR to Pronto hex and fires an event on the HA bus. This is an interim approach until HA ships `InfraredReceiverEntity` (expected 2026.6-2026.7), at which point the `ir_rf_proxy` RX entity will handle this natively.

## Mechanical Considerations

- **Form factor:** Small (roughly 40x40mm to 60x40mm). Should be unobtrusive on a wall or ceiling.
- **Enclosure:** Design for a 3D-printable snap-fit case, or a commercially available project box.
- **IR LED placement:** If multi-directional, LEDs should be on the board edge or on angled pads. Some designs use a cylindrical PCB with LEDs around the perimeter.
- **IR receiver placement:** Should have clear line-of-sight. Place on opposite side from TX LEDs to minimize self-interference.
- **Thermal:** IR LEDs at high duty cycle generate heat. Ensure adequate copper pour around the MOSFET and LED pads.


## Design Goals for Distribution

If the PCB is offered to others:

1. **Easy assembly** -- Use JLCPCB/LCSC-stocked parts. Prefer SMD components in 0805 or larger for hand-soldering compatibility. The ESP32 module is the only fine-pitch component.
2. **Single-sided assembly** -- Keeps PCBA costs down. Through-hole IR components (LED, receiver) can be on the opposite side.
3. **Standard USB-C** -- No proprietary connectors.
4. **Silkscreen labels** -- Mark TX, RX, USB, GPIO breakout pads clearly.
5. **Breakout pads** -- Expose a few spare GPIOs on pads or a header for user expansion (temperature sensor, relay, etc.).
6. **Test points** -- Add test points for 3.3V, GND, TX GPIO, RX GPIO for debugging.
7. **FCC/CE considerations** -- The ESP32-C3-MINI-1 module is pre-certified. Keep the antenna area clear of copper pour. Follow the module's reference design for antenna clearance.


## Software Integration Points

The PCB design doesn't need to worry about software beyond ensuring the GPIOs and power are correct. But for context:

- HAIR discovers ESPHome proxies via HA's device registry. Any ESPHome device with the `infrared` platform configured will appear in HAIR's Devices page under "Proxies" (RX) and "Emitters" (TX).
- The ESPHome firmware YAML is the only software that runs on the ESP32. It's flashed via USB initially, then updated OTA.
- HAIR's admin panel shows proxy devices with their friendly names and links to the ESPHome integration page.
- Multiple HAIR proxies can coexist -- one per room. They all report to the same HA instance.


## Open Questions for PCB Design

1. **Single vs. multi-frequency RX:** We currently assume 38kHz (covers ~90% of consumer IR). Some devices use 36kHz or 40kHz. The TSOP38238 has decent bandwidth to cover these, but a TSOP34838 (wider band) might be better for maximum compatibility.

2. **TX LED current:** Higher current = longer range but more heat. Typical: 100-200mA pulsed per LED. The MOSFET and current-limiting resistor set this. Need to decide target range (2m? 5m? 10m?).

3. **Power source:** USB-C only, or also support 5V barrel jack / PoE? USB-C is simplest for v1.

4. **Board panelization:** If ordering in quantity, design for panel breakaway to reduce per-unit cost.

5. **ESP32-C3 vs. ESP32-S3:** C3 is cheaper and sufficient. S3 adds Bluetooth (useful for BLE-based IR blasters like SwitchBot) and more GPIOs. Worth considering for v2.

6. **External IR blaster port:** A 3.5mm jack for an external IR cable is very useful for enclosed AV cabinets. Should this be on v1?
